## Story

**Como** aluno administrador da app FIFA 2026,
**eu quero** que a página `/standings` tenha um **seletor de fase** (Grupos · 32 avos · Oitavas · Quartas · Semis · 3º/4º · Final) e que **conforme eu encerre as partidas**, os times dos grupos avançam para o R32 e os vencedores cascateiem até a Final, **seguindo o bracket oficial FIFA 2026**,
**para que** o app simule a Copa inteira e os alunos vejam a árvore eliminatória sendo preenchida em tempo real conforme alimentam resultados.

### Estrutura oficial FIFA 2026 (do Annex C / Knockout Bracket)

**Round of 32 (matches 73-88):** mistura 1ºs, 2ºs e os 8 melhores 3ºs em cruzamento determinístico. Cada slot de "best 3rd" tem grupos elegíveis FIFA:

| Match | Side 1 | Side 2 |
|---|---|---|
| 73 | 2A | 2B |
| 74 | 1E | best 3rd ∈ {A,B,C,D,F} |
| 75 | 1F | 2C |
| 76 | 1C | 2F |
| 77 | 1I | best 3rd ∈ {C,D,F,G,H} |
| 78 | 2E | 2I |
| 79 | 1A | best 3rd ∈ {C,E,F,H,I} |
| 80 | 1L | best 3rd ∈ {E,H,I,J,K} |
| 81 | 1D | best 3rd ∈ {B,E,F,I,J} |
| 82 | 1G | best 3rd ∈ {A,E,H,I,J} |
| 83 | 2K | 2L |
| 84 | 1H | 2J |
| 85 | 1B | best 3rd ∈ {E,F,G,I,J} |
| 86 | 1J | 2H |
| 87 | 1K | best 3rd ∈ {D,E,I,J,L} |
| 88 | 2D | 2G |

**R16 (89-96):** 89=W74×W77, 90=W73×W75, 91=W76×W78, 92=W79×W80, 93=W83×W84, 94=W81×W82, 95=W86×W88, 96=W85×W87

**QF (97-100):** 97=W89×W90, 98=W91×W92, 99=W93×W94, 100=W95×W96

**SF (101-102):** 101=W97×W98, 102=W99×W100

**3º/4º (103):** L101 × L102 (perdedores das semis)

**Final (104):** W101 × W102

## Acceptance Criteria

### Backend

1. **AC-1** — Migration SQL `2026-05-07-knockout-matches.sql` cria 32 matches placeholder no DB com IDs sequenciais começando após o último id existente, com `stage` em ('round_of_32','round_of_16','quarter_final','semi_final','third_place','final'), `home_team_id`/`away_team_id` NULL, `status='scheduled'`, `home_score`/`away_score` NULL. Datas/horários genéricos (junho/julho 2026), stadium_id pode usar metlife (final stadium) como default.

2. **AC-2** — Endpoint `GET /api/bracket` retorna estrutura completa:
   ```json
   {
     "bracket": {
       "round_of_32": [{ "match_id": ..., "match_number": 73, "slot1": {...}, "slot2": {...}, "score1": null, "score2": null, "status": "scheduled" }, ...x16],
       "round_of_16": [...x8],
       "quarter_final": [...x4],
       "semi_final": [...x2],
       "third_place": {...},
       "final": {...}
     }
   }
   ```
   Cada `slot` é `{ team_id?: number, team_code?: string, team_name?: string, team_flag?: string, label: string }`. Quando team está definido, vem com dados; quando ainda é placeholder, `label` é "1º Grupo A", "2º Grupo B", "Melhor 3º (A/B/C/D/F)", "Vencedor M73", "Perdedor M101", etc.

3. **AC-3 (matching bipartite dos 3ºs)** — Quando os matches de TODOS os 12 grupos estão `finished`, o backend identifica os 12 terceiros, ranqueia por (points DESC, gd DESC, gf DESC, name ASC), pega top-8, e aloca em matching bipartite respeitando os grupos elegíveis FIFA por slot (mapa em código). Restrições satisfeitas: cada slot só recebe 3º de grupo elegível; cada 3º vai para exatamente 1 slot.

4. **AC-4 (cascade R32 → R16 → ...)** — Quando match do R32 tem `status='finished'` AND placar definido, a próxima fase recebe o vencedor (e perdedor para third_place no caso das semis). Backend calcula in-memory baseado em scores. Empate sem critério (penalties) NÃO suportado nesta story — admin define vencedor por placar (qualquer placar diferente).

5. **AC-5 (persistência opcional)** — Cada chamada de `/api/bracket`, o backend faz `UPDATE` nos matches do mata-mata cujos teams foram resolvidos pela propagação (e que ainda estão NULL no DB). Isso permite que `AdminMatches.tsx` (lista todos os matches) mostre teams resolvidos e admin possa encerrar via UI existente da Story 0.7.

### Frontend

6. **AC-6** — `Standings.tsx` ganha `<Tabs>` no topo (após o header). Valores: `groups`, `r32`, `r16`, `qf`, `sf`, `third_place`, `final`. Default: `groups`.
7. **AC-7** — Quando aba = `groups`: renderiza UI atual da Story 0.7 (12 cards de tabela compacta).
8. **AC-8** — Quando aba = mata-mata: renderiza grid de cards de confronto. Cada card mostra match_number, slot1 (label OU bandeira+código), placar (se finished), slot2, status. Top-down responsive: r32 em grid 2×8 ou 4×4; r16 em grid 2×4; qf em 2×2; sf+3rd+final em layout especial central.
9. **AC-9** — `lib/api.ts` ganha `getBracket()` retornando `Promise<ApiResponse<BracketResponse>>`. Hook `useQuery(['bracket'])` em `Standings.tsx` quando aba ≠ groups.
10. **AC-10** — Loading skeleton + error toast (mesmos padrões das outras stories).

### Cross

11. **AC-11** — Smoke test cascade: encerrar matches da fase de grupos (suficientes para definir top-2 de pelo menos um grupo) → /api/bracket retorna R32 com teams reais nos slots dos top-2 daquele grupo. Encerrar match do R32 → próxima fase (R16) recebe o vencedor.

> Snippet do bipartite matching (greedy + backtrack):

```js
const R32_THIRD_SLOTS = [
  { matchNum: 74, eligibleGroups: ['A','B','C','D','F'] },
  { matchNum: 77, eligibleGroups: ['C','D','F','G','H'] },
  { matchNum: 79, eligibleGroups: ['C','E','F','H','I'] },
  { matchNum: 80, eligibleGroups: ['E','H','I','J','K'] },
  { matchNum: 81, eligibleGroups: ['B','E','F','I','J'] },
  { matchNum: 82, eligibleGroups: ['A','E','H','I','J'] },
  { matchNum: 85, eligibleGroups: ['E','F','G','I','J'] },
  { matchNum: 87, eligibleGroups: ['D','E','I','J','L'] },
];

function matchThirdsToSlots(top8Thirds) {
  // top8Thirds: array de { team, group_name } ordenado por rank (melhor primeiro)
  // Retorna: Map<matchNum, third>
  const assignments = new Map();
  const slotsLeft = R32_THIRD_SLOTS.map(s => ({ ...s }));

  function backtrack(idx) {
    if (idx === top8Thirds.length) return true;
    const third = top8Thirds[idx];
    for (const slot of slotsLeft) {
      if (assignments.has(slot.matchNum)) continue;
      if (!slot.eligibleGroups.includes(third.group_name)) continue;
      assignments.set(slot.matchNum, third);
      if (backtrack(idx + 1)) return true;
      assignments.delete(slot.matchNum);
    }
    return false;
  }

  if (!backtrack(0)) {
    console.warn('Matching bipartite falhou — combinação dos 8 melhores 3ºs não tem alocação válida FIFA');
    return new Map(); // bracket sem 3ºs alocados
  }
  return assignments;
}
```

## Dev Notes

**Por que matching bipartite com backtrack?**
- Cada slot tem 5 grupos elegíveis, há 8 slots × 8 thirds = ate ~5^8 combinações. Backtrack é O(8!) worst case = 40320 — instantâneo.
- Garante alocação válida (nenhum 3º cai em slot inelegível). Diverge da tabela 495 FIFA só em casos de empate de mesma elegibilidade — aceito.

**Por que `persistResolvedTeams` é importante?**
- Se backend só calcula in-memory, os matches do mata-mata no DB ficam com home/away NULL.
- Isso impede admin de encerrar via UI (Story 0.7 desabilita botão Flag se teams NULL).
- Persistir os teams resolvidos permite admin USAR o existing flow.
- Trade-off: cada chamada de /api/bracket faz UPDATE — para 32 matches, máximo ~32 UPDATE statements por chamada. Aceitável para evento.

**Por que empate sem desempate (penalties)?**
- Adicionar penalties exige novo campo no schema (penalties_home, penalties_away) + UI de admin.
- Para evento, admin pode simplesmente colocar placar diferente (ex.: 2x1 em vez de 1x1). Documenta como "fase eliminatória não tem empate — admin define vencedor pelo placar regular".
- Fica como TD-11 para iteração futura.

**Sobre stadium dos matches mata-mata:**
- Migration usa MetLife como default (stadium da final real).
- Para alinhamento FIFA exato, seria preciso definir stadium específico por match — overkill para evento.
- TD-12: configurar stadiums corretos por match.

**Cache react-query:**
- Após admin encerrar match, `/api/bracket` precisa ser re-fetched para refletir cascade. Atual: usuário recarrega. TD-13: invalidate `['bracket']` quando admin update match.
